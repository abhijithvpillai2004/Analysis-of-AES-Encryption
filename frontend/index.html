AES-GCM — Authenticated Encryption

AES-GCM (Galois/Counter Mode) provides both confidentiality and data integrity in a single pass. It is the recommended mode for modern secure systems and is used in TLS 1.3, SSH, and HTTPS.

GCM extends CTR mode with a Galois field multiplier to produce an authentication tag (128-bit). If the ciphertext is tampered with, decryption will fail with an integrity error — even without knowing the key.

Observation
GCM produces three outputs: ciphertext, an IV/nonce, and an authentication tag. The tag is computed over both the ciphertext and the AAD.
Inference
Since GCM is built on CTR mode, it inherits CTR's parallelizability and eliminates the need for padding. The Galois authentication ensures tampered data is detected before any decryption occurs.
Conclusion
AES-GCM provides authenticated encryption with associated data (AEAD) — the gold standard for symmetric encryption. It solves the padding oracle and bit-flip attack vulnerabilities inherent in CBC mode.

Mode Reference

Technical reference for AES operation modes as defined in NIST FIPS 197 and NIST SP 800-38A.

ECB — Electronic Codebook
NIST SP 800-38A §6.1
⚠ Avoid

Each 16-byte block is encrypted independently. There is no diffusion between blocks — identical plaintext blocks always produce identical ciphertext blocks. This is the classic ECB penguin vulnerability where bitmap image structure remains recognizable after "encryption".

P₁ P₂ P₃ │ │ │ [AES_K] [AES_K] [AES_K] ← Same key, no mixing │ │ │ C₁ C₂ C₃ If P₁ == P₂ → C₁ == C₂ ← LEAKS structure!
PropertyValueSecurity implication
IV requiredNoNo randomness — same plaintext → same output always
Pattern leakageYesCritical structural weakness
ParallelizableYes (both)Fast but at the cost of security
PaddingPKCS7 requiredPotential padding oracle if not handled
Error propagationSingle blockErrors stay local — no cascading
Recommended useNever for dataOnly for encrypting single 16-byte keys
CBC — Cipher Block Chaining
NIST SP 800-38A §6.2
Standard choice

Each block is XORed with the previous ciphertext block before encryption, creating a chain. A random IV seeds the chain. Identical plaintexts encrypted with different IVs produce entirely different ciphertexts. Encryption is sequential; decryption can be parallelized.

IV C₀ C₁ │ │ │ └──→ XOR P₂→XOR │ │ [AES_K] [AES_K] │ │ C₁ C₂ C_{i} = AES_K(P_{i} ⊕ C_{i-1})
PropertyValueSecurity implication
IV requiredYes (16B)Must be unpredictable per message
Pattern leakageNoChaining prevents identical block outputs
Enc parallelizableNoSequential dependency — slower at scale
Dec parallelizableYesBlocks can be decrypted independently
PaddingPKCS7 requiredPadding oracle attacks (BEAST, POODLE)
Error propagation2 blocksA bit flip in C_i corrupts P_i and P_{i+1}
CTR — Counter Mode
NIST SP 800-38A §6.5
Best for streaming

Turns AES into a stream cipher. Encrypts a counter value (nonce ∥ counter) to generate a keystream, then XORs with plaintext. No padding needed. Both encryption and decryption are fully parallelizable and support random access to any block.

Nonce∥0 Nonce∥1 Nonce∥2 │ │ │ [AES_K] [AES_K] [AES_K] │ │ │ KS₀ KS₁ KS₂ │ │ │ P₀⊕ P₁⊕ P₂⊕ │ │ │ C₀ C₁ C₂ C_i = KS_i ⊕ P_i (stream cipher!)

IV requiredYes (nonce 8B)Never reuse (key, nonce) pair
Pattern leakageNoUnique counter per block
ParallelizableYes (both)Fastest mode at scale
PaddingNot requiredNo padding oracle risk
Random accessYesCan decrypt any block independently

Results & Conclusions

Consolidated findings from all experiments. Structured for viva presentation.

Experiment 1 ECB Pattern Leakage

We encrypted a plaintext containing 4 identical 16-byte blocks with ECB, CBC, and CTR modes and examined the ciphertext block frequencies.

Observation
In ECB mode, the 4 identical plaintext blocks produced 4 identical ciphertext blocks. In CBC and CTR mode, all ciphertext blocks were unique — no two blocks shared the same value.
Inference
ECB mode encrypts each block independently using the same key transformation. Since there is no state carried between blocks, the mapping P→C is deterministic and stateless. An adversary observing only the ciphertext can identify which blocks carry the same plaintext — this is a chosen-plaintext-equivalent information leak.
Conclusion
ECB mode is semantically insecure under the IND-CPA definition. It must never be used to encrypt data with any repeating structure. CBC and CTR both achieve semantic security through the use of a random IV/nonce.
Experiment 2 Performance Analysis

We measured encryption throughput across 64B, 256B, 1KB, and 4KB payloads at 50 iterations per size using AES-256.

Observation
CTR mode consistently achieved the highest throughput at larger data sizes. CBC was competitive at small sizes but showed a relative slowdown at 4KB+ due to its sequential encryption dependency. ECB's block-by-block Web Crypto implementation showed higher overhead.
Inference
CTR mode is parallelizable for both encryption and decryption. Its counter generation is independent per block, allowing hardware and software pipelines to process multiple blocks simultaneously. CBC's encryption step requires waiting for C_{i-1} before computing C_i — a fundamental sequential bottleneck.
Conclusion
CTR mode provides the best throughput for bulk encryption, particularly on multi-core systems. CBC encryption is inherently sequential, making it unsuitable for high-performance streaming applications. For large data, CTR or GCM should be preferred.
Experiment 3 AES-GCM Integrity

We encrypted a message with AES-GCM, then simulated a 1-bit tamper attack on the ciphertext and attempted decryption.

Observation
When the ciphertext was modified by even a single bit, decryption raised an OperationError — authentication tag mismatch. The original encryption with an untampered ciphertext decrypted successfully.
Inference
GCM's authentication tag is computed using GHASH over both the ciphertext and the associated data (AAD). Any modification to either the ciphertext or the context invalidates the tag. This provides ciphertext integrity — an attacker cannot modify the data without detection.
Conclusion
AES-GCM is an AEAD scheme (Authenticated Encryption with Associated Data). It prevents the padding oracle and bit-flip attacks that affect CBC mode. It is the recommended cipher suite in TLS 1.3 (RFC 8446) and is mandated for use in government systems (NIST SP 800-38D).
Final Summary — Mode Selection Guide
Mode Semantic Security Integrity Performance Recommendation
ECB No (IND-CPA fails) No Medium Never use
CBC Yes (with random IV) No (need MAC) Sequential enc Legacy systems
CTR Yes (with unique nonce) No (need MAC) Best With separate HMAC
GCM Yes Yes (auth tag) Best Modern standard
Project Conclusion: AES as a block cipher is secure across all modes when used correctly. The security difference lies entirely in the mode of operation. ECB's pattern leakage is a fundamental, unavoidable weakness. CBC provides security but is vulnerable to padding oracle attacks and has limited performance scaling. CTR and GCM represent the modern standard — offering parallelizability, padding-free operation, and (in GCM's case) built-in authentication. The principle of encrypt-then-authenticate should always be followed when using CBC or CTR without GCM.
